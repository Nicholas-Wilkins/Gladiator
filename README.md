# Gladiator

A self-improving chess engine that uses an evolutionary champion/challenger framework to continuously train itself. Two independent engines are included:

- **Gladiator** — CPU-only heuristic engine using alpha-beta negamax search with hand-tuned piece-square tables
- **Gladiator-NN** — GPU-accelerated engine using a small neural network (MLP) to evaluate board positions

Both engines follow the same training loop: random bots compete against a reigning champion, and the champion is replaced whenever a challenger wins. After 100 consecutive wins, the engine enters **mutation mode**, where new challengers are generated by perturbing the champion's parameters rather than sampling from scratch.

---

## Requirements

- Python 3.10+
- For Gladiator-NN with GPU acceleration: an NVIDIA GPU with CUDA support

---

## Installation

```bash
git clone https://github.com/yourusername/gladiator.git
cd gladiator
pip install -r requirements.txt
```

For the NN engine, also install PyTorch. Visit [pytorch.org](https://pytorch.org/get-started/locally/) and select your platform/CUDA version, or install the CPU-only version:

```bash
pip install torch
```

## Running

### CPU Engine

Single training session with dashboard:
```bash
python main.py
```

Parallel training across multiple workers (recommended):
```bash
python main_parallel.py --workers 4
```

Options:
```
--workers N      Number of parallel training workers (default: 4)
--db PATH        Main SQLite database path (default: gladiator.db)
--seed N         RNG seed for reproducibility
--no-ui          Headless mode (no dashboard)
--history        Print champion lineage and exit
--stats          Print recent match stats and exit
--max-games N    Stop after N total games (useful for testing)
```

### NN Engine

Single training session with dashboard:
```bash
python main_nn.py
```

Parallel training (recommended, 2 workers default due to VRAM):
```bash
python main_nn_parallel.py --workers 2
```

Options:
```
--workers N      Number of parallel training workers (default: 2)
--db PATH        Main SQLite database path (default: gladiator_nn.db)
--seed N         RNG seed
--no-ui          Headless mode
--cpu            Force CPU even if a GPU is available
--history        Print champion lineage and exit
--stats          Print recent match stats and exit
--max-games N    Stop after N total games
```

### Running both engines simultaneously

Open two terminal windows:
```bash
# Terminal 1
python main_parallel.py --workers 3

# Terminal 2
python main_nn_parallel.py --workers 2
```

## How it works

### Training loop

1. A **champion** is loaded from the database (or bootstrapped on first run).
2. A **challenger** is generated — either randomly (RANDOM mode) or by mutating the champion's parameters (MUTATION mode).
3. The two bots play a series of paired games (swapping colors each pair) until one has a statistically clear lead.
4. If the challenger wins, it becomes the new champion and is saved to the database.
5. After 100 consecutive wins, the engine switches to MUTATION mode and starts hill-climbing from the champion's parameters.

### CPU engine architecture

- Evaluation: piece values + piece-square tables + mobility/pawn/king heuristics (centipawns)
- Search: alpha-beta negamax, depth 2
- Parameters: piece values and PST weights, evolved per-bot

### NN engine architecture

- Network: `773 → 256 → 128 → 64 → 1` MLP with ReLU activations and tanh output
- Input: 12-plane one-hot board encoding (768 floats) + 5 auxiliary features = 773 floats
- Search: depth-1 negamax with batched GPU inference over all legal moves
- Parameters: full network weights (~960KB float32), mutated with Gaussian noise

### Parallel training

The parallel coordinators spawn independent worker subprocesses, each with its own SQLite database and RNG seed. Workers are resume-safe — restarting the coordinator picks up each worker from where it left off. The dashboard shows per-worker stats and a combined match total across all workers (including any previously stopped workers).

## Dashboard

Both engines include a Rich terminal dashboard showing:

- Generation, mode (RANDOM / MUTATION), total matches, uptime
- Longest win streak achieved
- Reigning champion details
- Current match in progress (live score)
- Live board position (last completed game)
- Recent match history

---

## Database

Training state is persisted in SQLite. The schema includes:

- `champion` — current champion's parameters and training state
- `champion_log` — lineage of all past champions
- `matches` — result of every match played
- `games` — individual game PGNs and metadata

Worker databases are stored as `gladiator_worker_N.db` / `gladiator_nn_worker_N.db` alongside the main database.
